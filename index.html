Method & known limitations
  • Correction is done in velocity space: v_ship(t) ≈ moving_average(v_raw, window = lap duration), then v_relative = v_raw − v_ship, then distance = ∫|v_relative| dt.
  • If the FIT file contains lap messages (lap button presses or auto-lap), each lap's actual duration is used as its own window — this handles laps that vary in length/time, instead of assuming a fixed period. Falls back to autocorrelation, or a manual override, when no usable lap messages exist.
  • Gaps in the recorded timestamps (pause/resume) split the activity into independent segments, each corrected separately, so a pause never gets bridged with fabricated interpolated motion or lets the moving average blend across it.
  • Assumes ship velocity is roughly constant across one lap — reasonable since a ship's course/speed changes slowly compared to a ~1–2 min lap. An optional Kalman pre-smoothing step can reduce GPS jitter before this assumption is applied.
  • A known real-world distance (signage, lap count) can be supplied to calibrate the algorithmic result with a simple scale factor.
  • This is a fun/educational exercise — the simplest ground truth is still just counting laps × posted lap distance from the ship's track signage.
